# uv

**The Python tool that replaces pyenv + venv + pip + pip-tools + virtualenv + conda.**

`uv` is a single Rust-built binary that manages everything Python: which interpreter you have, what venv a project uses, which packages are installed, and how a script runs. It is faster than pip by 10–100×, the workflow is consistent across projects, and it deletes the entire mental load of "wait, am I in the right env?"

This is replacing your conda/miniconda/pyenv setup. Once you commit, you can `brew uninstall miniconda` and never look back.

## Why you'll like it

- One tool, one mental model. No more "is it pyenv? conda? activate? deactivate? source bin/activate?"
- `uv run script.py` figures out which interpreter, which venv, which deps — and runs the thing. Reproducible.
- `uv sync` turns `pyproject.toml` + `uv.lock` into a working venv in seconds.
- Manages Python versions itself: `uv python install 3.12` downloads and pins it. No pyenv needed.
- Backwards-compatible "pip mode": `uv pip install requests` works inside any venv just like `pip`, but faster.
- Works alongside Homebrew Python without conflict.

## Install

If installed via the dotfiles Brewfile, no action. Otherwise: `brew install uv`.

## The mental model: two workflows

uv supports two ways of working. Pick one per project:

1. **Project mode** (`uv add`, `uv sync`, `uv run`) — you have a `pyproject.toml`. uv manages the venv and lockfile for you. **This is the default and the right answer for new code.**
2. **Pip mode** (`uv venv`, `uv pip install`) — you want a venv you manage manually, install whatever, no lockfile. Useful for ad-hoc work, replacing `python -m venv` + `pip`.

You will mostly want project mode. Pip mode is the escape hatch for legacy `requirements.txt` projects.

## The 80% you'll actually use

### Python interpreter management

| Command | What it does |
| --- | --- |
| `uv python list` | All Pythons uv knows about (installed + downloadable). |
| `uv python install 3.12` | Download and install Python 3.12 into uv's managed location. |
| `uv python install 3.11 3.12 3.13` | Multiple at once. |
| `uv python pin 3.12` | Pin this directory to 3.12 (writes `.python-version`). |

### Project mode (the default)

| Command | What it does |
| --- | --- |
| `uv init my-thing` | Scaffold a new project (creates `pyproject.toml`, `.python-version`, etc.). |
| `cd my-thing && uv add requests` | Add `requests` as a dep. Updates `pyproject.toml` and `uv.lock`, installs into `.venv/`. |
| `uv add --dev pytest ruff` | Dev-only dependency. |
| `uv remove requests` | Remove a dep. |
| `uv sync` | Make `.venv/` match `uv.lock` exactly. Run after `git pull`. |
| `uv lock` | Re-resolve `pyproject.toml` and update `uv.lock` without changing the venv. |
| `uv run script.py` | Run `script.py` using this project's venv. |
| `uv run python` | Drop into a REPL using this project's venv. |
| `uv run pytest` | Same — runs the venv's `pytest`. |

### Pip mode (legacy / one-off)

| Command | What it does |
| --- | --- |
| `uv venv` | Create `.venv/` in cwd, using the pinned Python. (`uv venv -p 3.11` to pick.) |
| `source .venv/bin/activate` | Activate it (same as plain venv). |
| `uv pip install requests` | Install into the active venv. |
| `uv pip install -r requirements.txt` | The drop-in replacement for `pip install -r`. |
| `uv pip freeze > requirements.txt` | Export pins. |
| `uv pip sync requirements.txt` | Make the venv exactly match the file (removes extras). |

## "Wait — is it `uv run python script.py`, `uv python run`, or just `python script.py`?"

Honest answer: there are three valid ways to run a Python script and they're all subtly different.

**Recommended (project mode):**
```bash
uv run script.py
```
That's it. uv figures out: which Python (from `.python-version`), which venv (from `pyproject.toml`), and runs the script with `.venv/bin/python`. **This is what you should default to.**

If your script needs args:
```bash
uv run script.py --flag value
```

You may also see these — they all work, here's when to use which:

| Form | When |
| --- | --- |
| `uv run script.py` | **Default. Project mode.** Uses the project's venv. |
| `uv run python script.py` | Same thing, more explicit. Useful when the script lacks a shebang or needs a flag like `python -O script.py`. |
| `uv run --python 3.12 script.py` | Force a specific Python, ignoring `.python-version`. |
| `python script.py` | Only correct if you've manually `source .venv/bin/activate`'d. Easy to get wrong. |
| `uv python run` | **Doesn't exist.** Wrong. Don't use. |
| `uv pip install` | Install a package into a venv. **Not** for running scripts. |

The TL;DR: type **`uv run`**, then whatever you want to run. That's the whole API.

## Gotchas / tips

- `uv run` will auto-create `.venv/` and install deps if it's missing. You almost never need to `uv sync` manually unless something is corrupted.
- `pyproject.toml` is the source of truth. `uv.lock` is the resolved/pinned snapshot — commit both.
- For scripts that aren't part of a project, you can use [PEP 723 inline metadata](https://peps.python.org/pep-0723/): put `# /// script\n# dependencies = ["requests"]\n# ///` at the top of `script.py` and `uv run script.py` will spin up a temp venv with `requests` for you. Magical for one-offs.
- You can keep using conda environments uv didn't create — but there's no benefit. Migrate to uv per project as you touch them.
- `uv tool install ruff` installs CLI tools (like pipx). `uvx ruff check .` runs without a permanent install. Replaces pipx.

## Try it now

```bash
# 1. Install a Python you don't have
uv python list
uv python install 3.12

# 2. Create a fresh project
mkdir /tmp/uv-demo && cd /tmp/uv-demo
uv init
ls -la                # see pyproject.toml, .python-version, src/uv_demo/...

# 3. Add a dep
uv add requests

# 4. Run a one-liner using the project's venv
uv run python -c 'import requests; print(requests.get("https://httpbin.org/uuid").json())'

# 5. Try inline-script metadata mode (no project at all)
cat > /tmp/scratch.py <<'PY'
# /// script
# dependencies = ["rich"]
# ///
from rich import print
print({"hello": "uv"})
PY
uv run /tmp/scratch.py

# 6. Drop into the project REPL
cd /tmp/uv-demo
uv run python

# 7. Pip-mode workflow against an existing requirements.txt
mkdir /tmp/uv-pip && cd /tmp/uv-pip
uv venv -p 3.12
source .venv/bin/activate
echo 'requests' > requirements.txt
uv pip install -r requirements.txt
python -c 'import requests; print(requests.__version__)'
deactivate
```
